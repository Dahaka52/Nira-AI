# План Решения TTS Проблем (после внешнего аудита)

Дата: 2026-03-04  
Контекст: текущий `qwen3` sidecar на Win11, локально, streaming в WS + `hw-audio-out-client`.

## 1. Что полезного взято из внешнего аудита

Подтверждено и полезно:
- Нужна строгая диагностика: отличить проблему генерации (сам WAV уже рваный) от проблемы транспорта/плеера.
- Возможны два разных класса дефектов:
  - щелчки/ломаная огибающая на границах чанков;
  - длинные паузы из-за underrun (генерация медленнее воспроизведения).
- Требуется jitter/prebuffer (уже частично внедрено, но нужно тюнинговать).
- Нужно избегать лишней нарезки/overhead при WS-передаче аудио.
- Нужен fallback-план (второй TTS модуль) на случай нестабильности Qwen3 streaming.

Сомнительно/не берем как факт без локальной проверки:
- Универсальные утверждения про «обязательный compile+cuda-graphs» для всех систем.
- Часть ссылок и цифр latency/RTF без воспроизводимого локального бенчмарка.

## 2. Что подтверждено локальными логами

- `qwen3_tts_server.log`:
  - frequent `cudagraph partition due to dynamic shape ops`;
  - warmup очень долгий (десятки секунд);
  - `first-audio latency` скачет (примерно 3s, 4s, 7s+).
- `backend`:
  - много RESPONSE jobs отменяется `Smart Barge-in` до завершения TTS;
  - в отдельных сессиях `TTS operation qwen3` занимает 10–30+ секунд.
- `audio_out`:
  - регулярный `Rebuffering: 0/...` => буфер пустеет, нет стабильного притока PCM.

Вывод: сейчас одновременно присутствуют:
1) низкая/нестабильная скорость генерации чанков;  
2) частые прерывания RESPONSE из STT/barge-in;  
3) лишний overhead передачи на WS уровне.

## 3. Цели исправления

- TTFS (first audible speech) <= 1.5s в типовой короткой реплике.
- Без длинных пауз внутри одной реплики (не более редких микропауз < 150ms).
- Отсутствие «залипания»/бесконечного ожидания чанков.
- Управляемый barge-in: не рушить ответ ложными срабатываниями.

## 4. План внедрения

## Этап A — Быстрая стабилизация (внедряем сразу)
1. Убрать лишний WS overhead:
- отправлять `audio_chunk` одним сообщением (без дополнительной нарезки base64 в `jaison.py`).

2. Уменьшить риск «зависаний» Qwen3:
- временно дефолт `use_compile=0` для sidecar (оставить compile как опцию A/B).
- сохранить `flash_attention_2` в Py3.12 окружении.

3. Снизить риск авто-отмены RESPONSE:
- ввести настраиваемый порог `barge_in_min_non_filler_words` (по умолчанию 2, для теста поставить 4).
- при ложных срабатываниях дать конфиг-режим, где interrupt только по stop-командам.

4. Снизить latency до первого звучания:
- переключить `chunker_sentence` в `sentence` (вместо `full`) для более ранней подачи первого TTS чанка.

Критерий этапа A:
- реплика 1–2 предложения проигрывается без полной «тишины после первых слов».

## Этап B — Диагностика «генерация vs транспорт» (обязательно)
1. Добавить debug-режим записи:
- склейка входящих PCM от sidecar в локальный WAV на 1 запрос.

2. Добавить метрики чанков:
- количество, средний интервал между чанками, максимум интервала, суммарная аудио-длительность.

3. Сверка:
- если WAV уже рваный -> проблема в генерации/склейке чанков;
- если WAV ровный, а realtime рваный -> проблема транспорта/плеера/буфера.

## Этап C — Качество стыков чанков
1. Если останутся щелчки/ломаный тон:
- добавить опциональный Hann crossfade на границе chunk-to-chunk (5–15ms) на стороне playback или sidecar emit layer.

2. Провести A/B:
- `overlap_samples = 0` vs небольшое overlap + crossfade.

## Этап D — Производительность и форки
1. A/B профили sidecar:
- profile `stable`: compile=0;
- profile `fast`: compile=1.

2. Если `stable` все еще плох:
- отдельная ветка интеграции `dffdeeq/reku` runtime (без ломки текущего pipeline API).

## Этап E — Fallback
1. Добавить второй TTS operation (`tts_active_id` switch) как backup.
2. Переключение только конфигом, без переписывания pipeline.

## 5. Порядок выполнения прямо сейчас

1) Этап A.1/A.2/A.3/A.4 (код + конфиг).  
2) Этап B (WAV/chunk metrics).  
3) Повторный живой прогон и сравнение логов.  
4) Решение по Этапу C/D на основе фактов.

## 6. Статус внедрения (текущая итерация)

Сделано:
- `jaison.py`: убрана дополнительная нарезка base64 (`chunk_buffer`) при WS-отправке `audio_chunk` (1 событие на 1 PCM-чанк).
- `jaison.py`: добавлен конфиг `microphone.barge_in_min_non_filler_words` (используется только для прерывания активного ответа).
- `config.yaml`:
  - `chunker_sentence.mode = "sentence"`
  - `operations.tts[qwen3].process.use_compile = 0` (stable baseline для A/B)
  - `microphone.barge_in_min_non_filler_words = 4`
- `qwen3.py`: добавлена телеметрия стрима в backend-лог:
  - `first_chunk_ms`
  - `max_gap_ms`
  - `chunks/bytes/total_ms/text_len`

Что проверить в следующем живом прогоне:
- Появился ли устойчивый поток (исчезли ли частые `Rebuffering: 0/...`).
- Значения `Qwen3 stream stats`:
  - если `max_gap_ms` большой (сотни/тысячи мс) -> узкое место генерации sidecar;
  - если `max_gap_ms` низкий, а звук рваный -> проблема на playback/route уровне.

Обновление (итерация после перехода на Base voice-clone, 2026-03-04):
- По логам найдено узкое место №1:
  - sidecar запускался очень долго из-за `compile + warmup` в voice_clone профиле (`warmup ~67s`).
- По логам найдено узкое место №2:
  - compile-path падал в рантайме (`storage data ptrs are not allocated in pool`), sidecar перезагружал модель на лету;
  - после этого `first_chunk_ms` резко рос, поток деградировал.
- По логам найдено узкое место №3:
  - `Qwen3 stream stats` показывает `rtf > 1` (часто 3-8), т.е. генерация медленнее realtime;
  - при этом в `audio_out` шло постоянное `Rebuffering: 0/...`.
- По логам найдено узкое место №4:
  - текущая логика rebuffer в плеере возобновляла воспроизведение с очень малого буфера, отсюда микрокуски и «дребезг».

Внесённые правки (текущий рефакторинг):
- `apps/hw-audio-out-client/main.py`:
  - переработан rebuffer-resume: теперь реально используется `rebuffer_ms`, а не крошечный фиксированный порог;
  - при долгом ожидании применяется «мягкое» понижение порога вместо мгновенного микро-резюма;
  - обновлены дефолты буферов под медленный стрим (`max/pre/rebuffer` больше).
- `apps/tts-qwen3-server/start_server.py`:
  - для `voice_clone` режима compile-path принудительно отключается (стабильный профиль на Win для Base).
- `src/utils/operations/tts/qwen3.py`:
  - увеличен `sidecar_read_chunk_bytes` (уменьшение WS overhead).
- `src/utils/jaison.py`:
  - добавлено распознавание коротких greeting-реплик (например, «привет») как сигнал к ответу.
- `configs/config.yaml`:
  - профиль стабилизации: `use_compile=0`, `warmup_on_start=0`, увеличенные буферы `audio_output`;
  - `chunker_sentence.mode=sentence` и уменьшен `t2t.max_length` для ограничения сверхдлинных TTS-ответов.

Что проверять в следующем прогоне:
- Startup backend: должен сократиться (без долгого compile+warmup).
- `qwen3_tts_server.log`: не должно быть runtime compile-fallback/reload.
- `Qwen3 stream stats`:
  - целевой `first_chunk_ms` стабильно < 2s на коротких репликах;
  - `rtf` должен снижаться по сравнению с предыдущим прогонам.
- `audio_out`:
  - вместо частых микро-рывков должны быть более длинные непрерывные участки звука;
  - если `rtf` всё ещё значительно >1, возможны паузы, но без «дребезга» по 0.1-0.2s.

## 7. Архитектура TTS v2 (под voice-clone Base)

Новый фокус:
- не пытаться «лечить» артефакты только буфером;
- снизить частоту границ чанков и не проигрывать частичные хвосты;
- ограничить длину одной TTS-реплики, чтобы не уходить в 30-100с генерации.

Решения:
1) Генерация:
- `voice_clone + Base`, `compile=0` (стабильность > нестабильный pseudo-fast path).
- профиль emission более «крупный» (`emit/decode` выше), чтобы снизить overhead.
- ограничение `max_text_chars` на стороне TTS operation.

2) Пайплайн текста:
- `chunker_sentence.mode=full` для одной непрерывной TTS-задачи на реплику (без повторных first-chunk штрафов).
- снижение `t2t.max_length` для контроля верхней границы длительности ответа.

3) Playback:
- не выдавать в устройство «кусок + нули», если буфера не хватает на callback block.
- держать хвост в буфере до достижения rebuffer-цели.
- `chunk_fade_ms=0` (исключить импульсные артефакты на плотных границах).

Критерий успеха v2:
- исчезают «пуки/писк/loop-noise» на границах;
- остаются только паузы, связанные с реальным дефицитом генерации;
- паузы становятся реже, а непрерывные отрезки длиннее.

## 8. Доп. диагностика и фикс (после лога 17:16-17:19)

Новая подтвержденная проблема:
- В `voice_clone` для коротких текстов фиксировались аномально длинные генерации:
  - пример: `text_len=12`, `audio_s=30.719`, `rtf=3.449`, `total_ms=105935`.
- Это означает runaway-декод (модель доходит до верхнего лимита кадров и тянет шумовой/артефактный хвост).

Новая подтвержденная проблема в playback:
- `hw-audio-out` мог застревать в `rebuffer`, если поток уже закончился, а в буфере остался хвост меньше rebuffer-цели.
- Симптом: бесконечные `Rebuffering` после завершения ответа и недоигранный хвост фразы.

Внедренные изменения:
1) Жёсткая защита от runaway в voice-clone:
- динамический лимит `max_frames` по длине текста (12Hz budget) на клиенте и в sidecar;
- cap для коротких реплик сильно ниже прежних 256/384 кадров.

2) Исправлен конец потока в playback:
- `hw-audio-out` теперь обрабатывает `response.status=success/cancelled/error` как `stream_end`;
- при `stream_end` хвост буфера доигрывается, вместо вечного ожидания `rebuffer`-порога.

3) Тюнинг latency/jitter:
- понижен `rebuffer_ms`, ускорен soft-resume;
- отключён chunk fade по умолчанию, чтобы убрать импульсные артефакты на плотном стриме.

4) Обновлен профиль voice_clone:
- `emit/decode`: `4/64`, `first_chunk`: `2/32/32`;
- `max_new_tokens` и `max_frames` снижены;
- включен dynamic frame budget.

## 9. Сводка по A/B тестам (18 прогонов) + фидбек прослушки

Объективно по двум `manifest`:
- Диапазон `RTF`: `3.508 .. 4.468`, среднее `3.777`.
- Лучшие по RTF:
  - `tts_ab_12_e4_d48_f1_24_24_mf80.wav` -> `RTF 3.508`
  - `tts_ab_04_e4_d48_f2_24_24_mf96_opt1.wav` -> `RTF 3.513`
  - `tts_ab_05_e6_d64_f3_32_32_mf128_opt1.wav` -> `RTF 3.523`
- Хуже всего:
  - `decode_window=40` (`tts_ab_16/17`) -> `RTF 4.30+`
  - `do_sample=true` (`tts_ab_18`) -> `RTF 4.271`
  - `use_optimized_decode=false` (`tts_ab_10`) -> `RTF 4.025`

Надежные зависимости из цифр:
- `use_optimized_decode=true` обязательно (иначе скорость хуже).
- `do_sample=false` для скорости и стабильности тембра.
- Лучший рабочий диапазон на текущем железе: `emit=3..4`, `decode=48..56`, `first_chunk=1/24/24`.
- Слишком маленький `decode_window=40` ускорения не дает, а RTF ухудшает.
- `overlap_samples=256` дал щелчки, пользы по RTF не дал.

Уточнение по качеству (по фидбеку прослушки):
- Паузы в ряде файлов были естественными между словами, не считаем их дефектом.
- Критичный дефект: варианты с `привет + шум/тишина` и редкие артефактные хвосты.

Вывод:
- Основная проблема realtime не в базовом качестве офлайн-генерации (оно приемлемо/хорошо),
  а в том, что при `RTF ~3.5+` поток PCM приходит слишком медленно для непрерывного живого воспроизведения.

## 10. Следующий профиль прогона (RTF squeeze, итерация #1)

Цель: снизить рваность в живом диалоге за счет более коротких TTS-реплик и профиля из лучших A/B.

Что применяем:
- TTS decode-профиль из лидеров: `e4/d48/f1/24/24`, `max_frames=80`, `max_new_tokens=112`.
- Жестче ограничиваем длину одной TTS-реплики (`max_text_chars`) для контроля хвостов.
- Аудиовыход переводим в более легкий realtime-буфер (меньше target rebuffer).
- Эксперимент маршрутизации GPU: TTS на `gpu_id=0` (RTX 5070), llama на `gpu_id=1` (RTX 5060), чтобы снять конкуренцию за ускорение TTS.

Ожидание:
- Если узкое место именно генерация Qwen3, должна снизиться частота `Rebuffering` и сократиться длинная «дробь» в live playback.
- Если RTF останется >3, корневая причина в runtime Qwen3 на текущем стеке, а не в буферах.

## 11. Новые факты (после прогона 18:47)

1) Почему `Rebuffering` спамил даже в idle:
- В `hw-audio-out-client` конец стрима определялся только по `response.status`, но WS события JAIson несут `finished=true/success=true`.
- Из-за этого `stream_end` не ставился и хвост буфера мог висеть в режиме rebuffer.
- Фикс внедрен: `finished=true` теперь тоже завершает поток; при WS-disconnect также принудительно ставится `stream_end`.

2) Что с генерацией:
- По live логам: `text_len=17`, `audio_s=1.918`, `total_ms=8898`, `rtf=4.639`, `max_gap_ms=1351`.
- Это подтверждает, что основной источник «кусочности» — не плеер, а медленный/рваный приток чанков из TTS.

3) Локальный sidecar smoke (voice_clone, 0.6B Base):
- Даже при `use_optimized_decode=true` получено `RTF ~3.7+` в чистом запросе (без полного диалога).
- Следствие: текущий runtime ceiling для voice_clone на этом стеке заметно выше realtime.

4) Подготовлен controlled A/B для compile в voice_clone:
- Добавлен флаг `allow_compile_voice_clone` (по умолчанию `0` для стабильности).
- Теперь можно включать `use_compile=1` для voice_clone без ручного патча кода и сравнивать RTF/стабильность.
