# Архитектура Project J.A.I.son — Референс на русском

> **Проект:** jaison-core — серверный бэкенд для AI-компаньона/vTuber  
> **Лицензия:** MIT  
> **GitHub:** https://github.com/limitcantcode/jaison-core

---

## 1. Что это такое

**J.A.I.son Core** — это Python-сервер, выступающий **бэкендом** для AI-персонажа.  
Он не делает UI, не запускает VTuber-модель — он принимает аудио/текст, обрабатывает через LLM, возвращает текст + аудио.

Внешние приложения (Discord-бот, VTube Studio, Twitch-бот) общаются с ним по **REST API + WebSocket**.

### Ключевые принципы:
- **Асинхронность** (asyncio) — всё неблокирующее
- **Модульность** — любой компонент (STT/TTS/LLM) заменяется без изменения кода
- **Очередь задач (Job Queue)** — запросы выполняются строго по порядку
- **Event-driven** — результаты транслируются через WebSocket в реальном времени

## 3. Главный пайплайн ответа (response pipeline)

Это **центральная цепочка** работы системы. Запускается при вызове `POST /api/response`.

### Шаги пайплайна (`jaison.py → response_pipeline()`):

```
[Внешнее приложение]
       │
       │  REST: POST /api/response
       ▼
[Очередь задач (Job Queue)]
       │
       ▼
 ┌─────────────────────────────────────────────────────────┐
 │                    response_pipeline()                   │
 │                                                         │
 │  1. MCP (если активен)                                  │
 │     ├─ LLM-for-tools генерирует tool_calls              │
 │     └─ Результаты добавляются в историю                 │
 │                                                         │
 │  2. Prompter собирает промпт:                           │
 │     ├─ instruction_prompt (системные инструкции)        │
 │     ├─ character_prompt (личность персонажа)            │
 │     ├─ scene_prompt (сцена)                             │
 │     └─ history (история диалога, последние N сообщений) │
 │                                                         │
 │  3. T2T (LLM) → стримит токены → WebSocket             │
 │     Вход: {instruction_prompt, messages: [history]}     │ 
 │     Выход: {content: "текст", tps, latency}             │
 │                                                         │
 │  4. Text Filters (цепочка фильтров):                   │
 │     filter_clean → chunker_sentence → emotion_roberta   │
 │     Вход: {content: "текст"}                            │
 │     Выход: {content: "чистый текст", emotion?: "..."}   │
 │                                                         │
 │  5. Текст добавляется в историю як ответ персонажа      │
 │                                                         │
 │  6. TTS → аудио-чанки                                   │
 │     Вход: {content: "текст", emotion?: "..."}           │
 │     Выход: {audio_bytes, sr, sw, ch}                    │
 │                                                         │
 │  7. Audio Filters (цепочка):                           │
 │     pitch → rvc                                         │
 │     Вход: {audio_bytes, sr, sw, ch}                     │
 │     Выход: {audio_bytes, sr, sw, ch}                    │
 │                                                         │
 │  8. Аудио кодируется base64 → WebSocket               │
 └─────────────────────────────────────────────────────────┘
       │
       ▼
[WebSocket событие: audio_bytes]
```

### Как добавить ввод пользователя перед вызовом `/api/response`:

**Текстом:**  
`POST /api/context/conversation/text` с `{user, timestamp, content}`

**Голосом (аудио):**  
`POST /api/context/conversation/audio` с `{user, timestamp, audio_bytes(base64), sr, sw, ch}`  
→ STT автоматически транскрибирует аудио в текст и добавляет в историю

## 4. Типы сообщений в истории (message.py)

История диалога хранится как список объектов `Message`. У каждого типа свой формат строки:

| Тип | Python-класс | Формат в промпте |
|-----|-------------|-----------------|
| Чат-сообщение (пользователь/AI) | `ChatMessage` | `[HH:MM] Имя: текст` |
| Системный запрос/задача | `RequestMessage` | `[request]: текст` |
| Результат MCP-инструмента | `MCPMessage` | `[tool][инструмент]: результат` |
| Пользовательский контекст | `CustomMessage` | `[имя_контекста]: текст` |
| Сырой текст (без форматирования) | `RawMessage` | `текст` |

**Формат истории в WebSocket-событии:**
```json
{
  "history": [
    {"type": "raw", "message": "..."},
    {"type": "request", "time": 1234567890, "message": "..."},
    {"type": "chat", "time": 1234567890, "user": "Username", "message": "..."},
    {"type": "tool", "time": 1234567890, "tool": "tool_name", "message": "..."},
    {"type": "custom", "time": 1234567890, "id": "context_id", "message": "..."}
  ]
}
```

### 6.7 MCP (Model Context Protocol) — Внешние инструменты
- LLM с ролью `mcp` читает инструменты и вызывает их
- Результаты добавляются в историю как `MCPMessage`
- Конфигурируется в YAML:
```yaml
mcp:
  - id: my_server
    command: python
    args: ["my_mcp_server.py"]
    cwd: "path/to/server"
```

Это важно понимать — следующие функции в базовом проекте **отсутствуют**, их нужно разрабатывать отдельно:

| Функция | Статус | Где зацепиться |
|---------|--------|---------------|
| Долгосрочная память (векторная БД) | ❌ Нет | `embedding` + кастомный контекст |
| Рефлексивная память / self-reflection | ❌ Нет | Новый MCP-сервер или операция |
| Идентификация по голосу | ❌ Нет | Новая операция перед STT |
| Дообучение (fine-tuning) | ❌ Нет | Внешние скрипты + `history.txt` |
| Развитие характера со временем | ❌ Нет | `character_prompt` + внешняя логика |
| Интеграция с Discord | 🔗 Внешнее приложение | [app-jaison-discord-lcc](https://github.com/limitcantcode/app-jaison-discord-lcc) |
| Интеграция с VTube Studio | 🔗 Внешнее приложение | [app-jaison-vts-hotkeys](https://github.com/limitcantcode/app-jaison-vts-hotkeys-lcc) |
| Интеграция с Twitch/стримы | 🔗 Внешнее приложение | [app-jaison-twitch](https://github.com/limitcantcode/app-jaison-twitch-lcc) |
| Несколько голосов/пользователей | ⚠️ Частично | `user` параметр в истории |
| Эмоции в ответах | ✅ Есть | `emotion_roberta` фильтр |
| Голосовой фильтр/клон голоса | ✅ Есть | `rvc` audio filter |

## 11. Как создать новую операцию (пример)

```python
# src/utils/operations/stt/my_stt.py
from ..base.stt import BaseSTT

class MySTT(BaseSTT):
    def __init__(self):
        super().__init__("my_stt_id")  # уникальный id
    
    async def start(self):
        await super().start()
        # инициализация клиента
    
    async def close(self):
        await super().close()
        # закрытие соединений
    
    async def _generate(self, audio_bytes, sr, sw, ch, prompt, **kwargs):
        # ... обработка ...
        yield {"transcription": "распознанный текст"}
```

Затем добавить импорт в `operations/manager.py` в функцию `load_op()`.

## 13. Что уже сделано с исходным проектом

Это список изменений, которые были внесены поверх оригинального jaison-core:

### ✅ Переход на llama.cpp
- Вместо KoboldCPP в качестве T2T-бэкенда используется **llama.cpp**
- Операция `llamacpp` или `llamacpp_python` указана в конфиге как `role: t2t`
- Модели в формате GGUF загружаются напрямую через llama.cpp без дополнительного сервера
- Параметры в `config.yaml`: `llamacpp_filepath`, `llamacpp_model_filepath`, `llamacpp_ctx`, `llamacpp_ngl` (кол-во слоёв на GPU)

### ✅ Начальная версия GUI через браузер
- Реализован браузерный интерфейс для общения с персонажем
- GUI взаимодействует с сервером через REST API и WebSocket
- Пользователь может вводить текст и получать ответы без внешних приложений

### ✅ Базовая работа модели
- Модель успешно запускается и генерирует ответы
- Пайплайн T2T → filter_text → TTS работает в связке
- Стриминг токенов подтверждён

### ✅ Архитектурные исправления стабильности (Process Management & Routing)
- **Динамический биндинг портов**: `LlamaCPPT2T` больше не кэширует порт `llama-server` при инициализации. Порт запрашивается динамически у `ProcessManager` при каждом запросе (`_generate`). Это навсегда закрывает проблему "All connection attempts failed" после крашей или рестартов фонового процесса (когда сервер поднимается на новом случайном порту, а пайплайн пытается стучаться в старый мёртвый порт).
- **Ожидание готовности (Polling)**: При запуске `llama-server` добавлено активное ожидание `HTTP GET /health` (до 120 секунд). Теперь пайплайн не отправляет запросы в LLM, пока модель полностью не загрузится в VRAM.
- **Стабильное логирование (Абсолютные пути)**: Все отладочные логи (`debug_gui.log`) теперь пишутся по абсолютному пути через `args.log_dir`, что решает проблему потери логов при запуске бэкенда через фоновые Job-ы PowerShell, где `os.getcwd()` может указывать в другое место.
- **Изоляция и запуск процессов**: Скрипты запуска переведены на PowerShell 7 (`pwsh`) для корректной поддержки `-WorkingDirectory` в `Start-Job`. *Важное примечание:* `Stop-Job` не закрывает дочерние процессы (Python, llama-server). При ошибках занятых портов нужно вручную завершать "зомби" процессы `python.exe`.
- **Чистота REST API**: Устранены конфликты роутинга в `app_server.py` (удалены дублирующиеся POST/PUT эндпоинты типа `/api/config/update`), добавлен независимый `GET /api/health` для проверки жизнеспособности бэкенда.

### 🧱 Что такое «Операция»

**Операция** — это отдельный «модуль-инструмент», который умеет делать одно конкретное дело.  
Например: распознать речь, сгенерировать текст, озвучить текст.

Каждая операция:
- **Получает** данные на вход (словарь с полями)
- **Обрабатывает** их (делает API-запрос, запускает ML-модель, применяет фильтр)
- **Возвращает** данные на выход (тоже словарь с полями)

Операции **не знают друг о друге** — они как конвейерные станции. Данные просто передаются от одной к другой.

---

### 📦 Что такое «Словарь» / «Поля»

**Словарь** (в контексте Python и этого проекта) — это набор пар **«ключ: значение»**.  
В JSON-виде выглядит так:
```json
{
  "content": "Привет, как дела?",
  "emotion": "joy",
  "audio_bytes": "...байты...",
  "sr": 22050
}
```

Каждый элемент в словаре — это **поле**.

| Понятие | Простое объяснение |
|---------|-------------------|
| **Поле** | Одна конкретная именованная переменная в словаре. Например `content` — это поле |
| **Значение поля** | То, что хранится в нём. Например поле `content` содержит строку текста |
| **Ключ** | Имя поля. То, по чему его ищут — например `"content"`, `"sr"`, `"audio_bytes"` |
| **Словарь (dict)** | Весь набор полей вместе — весь «пакет данных» передаваемый между операциями |
| **Чанк** | Один маленький кусок данных. LLM не ждёт пока напишет всё — она шлёт по токену за раз. Каждый такой кусок — это чанк |

---

### 🔁 Что такое «Стриминг»

**Стриминг** — это когда данные возвращаются **не разом**, а по чуть-чуть, по мере готовности.

Пример: LLM не ждёт пока напишет весь ответ, а отправляет слово за словом:
```
чанк 1: {"content": "При"}
чанк 2: {"content": "вет"}
чанк 3: {"content": ", как"}
чанк 4: {"content": " дела?"}
```

Это снижает задержку — пользователь видит/слышит ответ быстрее.  
В коде стриминг реализован через `async for chunk in operation(...)`.

---

### 📡 Что такое «WebSocket» и «REST API»

| Понятие | Простое объяснение |
|---------|-------------------|
| **REST API** | Обычные HTTP-запросы. Ты стучишься на адрес (URL), получаешь ответ. Как открыть страницу в браузере, только данные |
| **Эндпоинт** | Конкретный URL, на который можно сделать запрос. Например `POST /api/response` |
| **WebSocket** | Постоянное двустороннее соединение. Сервер сам может присылать сообщения клиенту в любой момент — не нужно каждый раз спрашивать |
| **Событие (Event)** | Сообщение, которое сервер присылает через WebSocket. Например «вот кусок аудио» или «задание завершено» |
| **Job (задание)** | Запрос, который попадает в очередь и выполняется асинхронно. Ты создаёшь задание через REST, а результат получаешь через WebSocket |
| **Job Queue (очередь)** | Список заданий, ожидающих выполнения. Система берёт по одному — строго по порядку |
| **job_id** | Уникальный номер (UUID) задания. По нему можно отследить все события именно этого задания |

---

### ⛓️ Что такое «Пайплайн»

**Пайплайн** (pipeline, конвейер) — это **цепочка операций**, где выход одной является входом следующей.

```
Голос пользователя
      ↓
    [STT] → текст
      ↓
    [T2T / LLM] → текст-ответ
      ↓
    [filter_text] → очищенный/размеченный текст
      ↓
    [TTS] → аудио
      ↓
    [filter_audio] → обработанное аудио
      ↓
Воспроизведение
```

Каждый блок — отдельная **Операция**. Порядок задаётся в `config.yaml`.

---

### 🗂️ Типы операций и с чем они работают

#### STT (Speech-to-Text) — «Уши»
- **Получает:** сырые байты аудио + параметры звука
- **Делает:** распознаёт речь — голос превращает в текст
- **Отдаёт:** строку с распознанным текстом
- **Взаимодействует с:** микрофоном/аудио-входом приложения → передаёт текст в историю

```
Вход:  { audio_bytes, sr (sample rate), sw (sample width), ch (channels), prompt }
Выход: { transcription: "текст что сказали" }
```

> `sr` — частота дискретизации (обычно 16000 или 44100 Гц)  
> `sw` — ширина сэмпла в байтах (обычно 2)  
> `ch` — количество каналов (1 = моно, 2 = стерео)  
> `prompt` — подсказка для Whisper, помогает распознавать правильно (обычно кусок истории)

---

#### T2T (Text-to-Text / LLM) — «Мозг»
- **Получает:** системный промпт + историю диалога
- **Делает:** генерирует ответ — это и есть языковая модель (LLM)
- **Отдаёт:** токены текста стримом
- **Взаимодействует с:** Prompter (берёт промпт и историю) → передаёт текст в фильтры

```
Вход:  { instruction_prompt: "системный текст с правилами и личностью",
          messages: [список сообщений истории] }
Выход: { content: "кусок текста ответа" }  ← стримится по токену
```

> `instruction_prompt` — это весь системный промпт целиком: инструкции + персонаж + сцена + описания контекстов  
> `messages` — список объектов истории (ChatMessage, RequestMessage итд) — вся «память» разговора

---

#### filter_text — «Редактор текста»
- **Получает:** текст от LLM
- **Делает:** убирает мусор, определяет эмоцию, режет на предложения
- **Отдаёт:** чистый текст + дополнительные поля (например `emotion`)
- **Взаимодействует с:** T2T (берёт сырой ответ) → TTS (передаёт обработанный)

```
Вход:  { content: "текст от LLM" }
Выход: { content: "чистый текст", emotion: "joy" }  ← emotion появляется если подключён emotion_roberta
```

**Фильтры по порядку:**
| Фильтр | Что делает |
|--------|-----------|
| `filter_clean` | Убирает артефакты: лишние звёздочки, кавычки, теги, мусор от LLM |
| `emotion_roberta` | Добавляет поле `emotion` — определяет настроение текста |
| `chunker_sentence` | Режет текст на предложения — чтобы TTS начинала озвучивать сразу, не ждав весь текст |
| `mod_koala` | Проверяет на запрещённый контент (можно отключить) |

---

#### TTS (Text-to-Speech) — «Голос»
- **Получает:** текст (и опционально эмоцию)
- **Делает:** синтезирует аудио из текста
- **Отдаёт:** байты аудио стримом
- **Взаимодействует с:** filter_text (берёт текст) → filter_audio (передаёт сырое аудио)

```
Вход:  { content: "текст для озвучки", emotion: "joy" }
Выход: { audio_bytes: b"...", sr: 22050, sw: 2, ch: 1 }
```

---

#### filter_audio — «Звуковой процессор»
- **Получает:** сырое аудио от TTS
- **Делает:** меняет тон голоса, применяет эффекты, конвертирует в другой тембр (RVC)
- **Отдаёт:** обработанное аудио в том же формате
- **Взаимодействует с:** TTS (входное аудио) → WebSocket (отправляет клиенту)

```
Вход:  { audio_bytes: b"...", sr, sw, ch }
Выход: { audio_bytes: b"...", sr, sw, ch }  ← тот же формат, другое содержимое
```

---

#### embedding — «Переводчик смысла»
- **Получает:** строку текста
- **Делает:** превращает текст в вектор чисел (список из ~1500 чисел), который кодирует **смысл** текста
- **Отдаёт:** вектор (массив float-чисел)
- **Взаимодействует с:** системой памяти (если есть) — для поиска похожего по смыслу

> Зачем? Два похожих по смыслу текста дают похожие векторы. Это позволяет искать «воспоминания» не по ключевому слову, а по смыслу. В базовой версии проекта не используется.

---

#### MCP (Model Context Protocol) — «Руки / Инструменты»
- **Получает:** промпт с описанием доступных инструментов + текущий контекст разговора
- **Делает:** отдельная LLM решает — какой инструмент вызвать и с какими аргументами
- **Отдаёт:** результаты вызовов, которые добавляются в историю как `[tool]` сообщения
- **Взаимодействует с:** внешними MCP-серверами (Python-скрипты в `models/mcp/`) → результаты идут в Prompter

> Пример: MCP-сервер для интернета. LLM видит инструмент `search(query)`, решает его вызвать, получает результат, и **только потом** основная LLM генерирует ответ уже зная что нашлось в интернете.

---

### 🧠 Что такое «Prompter» и «История»

**Prompter** — это «память на время разговора» и «сборщик промптов».

Он хранит:
- Файлы с инструкцией, персонажем, сценой
- Историю диалога (последние N сообщений)
- Метаданные кастомных контекстов

Перед каждой генерацией Prompter собирает всё это в один большой текст — **системный промпт** — который и отправляется в LLM.

**История** — это список сообщений разговора. Каждое сообщение имеет тип, автора, время и текст. Хранится в памяти и дублируется в файл `output/history.txt`.

---

### 🔌 Что такое «Синглтон»

**Синглтон** — это объект, который существует в системе **только в одном экземпляре**.

В проекте синглтонами являются: `JAIson`, `Config`, `Prompter`, `OperationManager`, `ProcessManager`.

Это значит: где бы в коде ты ни написал `Config()` — ты всегда получишь **один и тот же объект** с одними и теми же данными. Нет риска что где-то будет «другая копия конфига».

## 17. Подготовка к Streaming STT и сменным STT-модулям

Этот раздел фиксирует минимальный контракт, чтобы можно было подключать разные STT без переписывания пайплайна.

### 17.1 Диалоговая state-machine (phase-2 baseline)

Состояния:
- `idle` -> нет активного ввода/вывода
- `listening` -> идёт пользовательская речь
- `thinking` -> LLM генерирует ответ
- `speaking` -> TTS/аудио выводится
- `interrupted` -> активный `response` отменён (stop/значимая реплика)

Переходы (минимум):
- `idle -> listening`: speech_start
- `listening -> thinking`: commit user turn
- `thinking -> speaking`: первый TTS chunk
- `thinking|speaking -> interrupted`: stop/interrupt policy
- `interrupted -> listening`: пользователь продолжает говорить
- `interrupted -> idle`: очередь очищена, новых реплик нет

Приоритеты intent в phase-2:
1. `stop_cmd`
2. `continue`
3. `new_topic`
4. `backchannel/filler`

### 17.2 Контракт STT-адаптера (обязательный)

Любой STT-модуль должен:
- принимать PCM байты + `sr/sw/ch`;
- возвращать финальный текст в формате `{ transcription: "..." }` (совместимость текущего пайплайна);
- опционально поддерживать partial-ивенты (для streaming phase-2) с полями:
  - `turn_id`
  - `utterance_id`
  - `text`
  - `is_final`
  - `confidence` (опционально)
  - `timestamp_ms` (опционально)

Требование совместимости:
- финальный `transcription` обязателен всегда;
- partial не должен ломать текущий `process_audio_immediate()`.

### 17.3 Переключение STT только конфигом (без переписывания пайплайна)

Добавлен селектор:
- `stt_active_id: "..."`

Правило загрузки:
- в `operations` можно держать несколько `role: stt`;
- `OperationManager` загружает только запись, чей `id == stt_active_id`;
- переключение: изменить `stt_active_id` + вызвать `POST /api/operations/reload`.

Пример:
```yaml
stt_active_id: "sherpa"
operations:
  - role: stt
    id: sherpa
    ws_url: "ws://localhost:6006"
    provider: "cpu"
  - role: stt
    id: openai
    model: "whisper-1"
    language: "ru"
```

Это позволяет быстрый A/B тест STT-модулей без изменений `jaison.py` и без правок остального пайплайна.
