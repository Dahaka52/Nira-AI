# Архитектура Nira (текущее состояние и целевой TTS-тракт)

Дата: 2026-03-04  
Репозиторий: `C:\Nirmita`

## 1. Назначение

Nira сейчас это локальный асинхронный backend для AI-персонажа на базе `jaison-core` с Web API, WebSocket-событиями, job-очередью и модульными операциями (`stt/t2t/filter_* /tts`).

Ключевой акцент текущего этапа: стабильный voice turn-taking + подготовка к подключению реального TTS как отдельного плагина.

## 2. Фактическое состояние репозитория

## 2.1 Что реально реализовано

1. Асинхронный backend на Quart + WebSocket и централизованный `JAIson` application layer.
2. Очередь задач (`job_queue`) с контролем отмены, skip queued jobs и корректной очисткой coroutine.
3. Immediate voice path вне очереди: `POST /api/context/conversation/audio` сразу уходит в `process_audio_immediate()` для быстрых прерываний.
4. Early barge-in endpoint: `POST /api/context/conversation/speech_start`.
5. STT в прод-конфиге: `sherpa` (через локальный websocket сервер на `:6006` + валидация/санитизация hotwords).
6. T2T в прод-конфиге: `llamacpp` (локальный `llama-server`, отдельный managed process, streaming tokens).
7. Text filters в прод-конфиге: `filter_clean` + `chunker_sentence`.
8. Метрики стриминга уже встроены: `ttft_ms`, `e2e_ttft_ms`, `tps`, `tts_start_ms`, `e2e_tts_start_ms`.
9. UI (`apps/nira-web`) получает и визуализирует текстовый стрим и метрики.
10. Микрофонный клиент (`apps/hw-mic-client`) с локальным VAD + пре-роллом + early speech_start.

## 2.2 Что есть в коде, но не активно в текущем конфиге

1. `emotion_roberta` фильтр (`SamLowe/roberta-base-go_emotions`).
2. Audio filters: `pitch`, `rvc`.
3. MCP manager и embedding operation (OpenAI-compatible).

## 2.3 Что отсутствует прямо сейчас

1. Реальный TTS модуль не подключен: в `OperationManager.load_op()` ветка `OpTypes.TTS` сейчас всегда кидает `UnknownOpID`.
2. В `configs/config.yaml` нет активной операции `role: tts`.
3. Веб-клиент не воспроизводит `audio_chunk` (обрабатывает текст/метрики).
4. В этом репозитории нет приложений Discord/Twitch/VTube Studio (из прикладных модулей есть `hw-mic-client`, `nira-web`, `stt-sherpa-server`; `stt-sensevoice-server` присутствует как заготовка без рабочего исходника).
5. Долгосрочная память (vector DB + retrieval loop) не реализована.

## 3. Ключевые тракты данных

## 3.1 Голосовой вход (уже рабочий)

`hw-mic-client` (VAD, pre-roll, short interrupt logic)  
-> `POST /api/context/conversation/speech_start` (ранний сигнал)  
-> `POST /api/context/conversation/audio` (base64 PCM)  
-> `JAIson.process_audio_immediate()`  
-> `STT(sherpa)`  
-> intent классификация (`stop/backchannel/continue/wake/emote/significant`)  
-> при необходимости `interrupt` текущего response  
-> буферизация/склейка user turn  
-> постановка jobs `context_conversation_add_text` + `response`.

## 3.2 Текстовый вход

`POST /api/context/conversation/text`  
-> запись в историю Prompter  
-> `POST /api/response`  
-> очередь jobs.

## 3.3 Тракт ответа (текущий)

`response_pipeline()`:

1. Проверка `include_audio` (автовыключение, если TTS не загружен).
2. MCP шаг (если есть роль `mcp`).
3. Сбор system prompt + history через Prompter.
4. T2T streaming (`OpRoles.T2T`) -> WS чанки текста.
5. Guard на пустой T2T результат (без падения).
6. FILTER_TEXT chain (`OpRoles.FILTER_TEXT`) по полному тексту.
7. Если TTS активен: для каждого text chunk  
`TTS -> FILTER_AUDIO chain -> base64 chunk_buffer -> WS event audio_chunk`.
8. Сохранение финального ответа в историю.

Семантически тракт для TTS уже готов, но провайдер TTS отсутствует.

## 4. Очередь, прерывания и консистентность

1. Job-порядок сохраняется через `asyncio.Queue`.
2. `cancel_job()` корректно работает и для running, и для queued задач.
3. `_interrupt_jobs()` очищает очередь и закрывает queued coroutine (без утечек/варнингов).
4. `speech_start_interrupt_mode`: `soft|hard`.
5. Voice merge/debounce/min quiet окна конфигурируются в `microphone` секции.
6. Есть continue-after-interrupt через `continue_from_text` в `response_pipeline()`.

## 5. Операционная модель (как подключать модули)

Единый контракт операций:

1. Каждая операция наследует `Operation` и реализует `configure/get_configuration/_parse_chunk/_generate`.
2. Runtime интерфейс: `async for chunk in operation(payload)`.
3. Реестр и загрузка модулей централизованы в `OperationManager`.
4. Фильтры (`filter_text`, `filter_audio`) поддерживают цепочку в порядке конфига.
5. Горячая перезагрузка: `POST /api/operations/reload`.

Это уже правильная база для TTS как plug-in, где добавляется только новый класс + регистрация в `load_op` + запись в `config.yaml`.

## 6. Целевой TTS-тракт (желаемая реализация)

## 6.1 Цели

1. TTS должен быть отдельной `operation` с `role: tts`, не вшитой в `jaison.py`.
2. Полная совместимость с текущим response pipeline.
3. Стриминг PCM чанков с минимальной задержкой.
4. Поддержка `emotion` (если поле пришло из text filter).
5. Переключаемость провайдера через YAML + `/api/operations/reload`.

## 6.2 Контракт TTS операции

Входной chunk:

```json
{ "content": "текст", "emotion": "опционально" }
```

Выходной stream chunk:

```json
{ "audio_bytes": "bytes", "sr": 24000, "sw": 2, "ch": 1 }
```

Требования:

1. `_generate()` не блокирует event loop надолго (тяжёлые куски через thread/executor, если нужно).
2. На пустом/невалидном тексте возвращается пустой stream без exception flood.
3. Формат аудио стабилен в пределах одной операции.

## 6.3 Цепочка после внедрения

`T2T chunk` -> `filter_text` (чистка/эмоция/чанкинг) -> `TTS(chunk)` -> `filter_audio(optional)` -> `WS audio_chunk` -> клиент/роутинг.

## 6.4 Конфигурация

Минимальный пример:

```yaml
operations:
  - role: tts
    id: qwen3
    model_id: "Qwen/Qwen3-TTS-0.6B"
    device: "cuda"
    sample_rate: 24000
    voice: "nira"
    stream_chunk_ms: 80
```

## 6.5 Наблюдаемость и SLA

1. Использовать уже существующие `tts_start_ms` / `e2e_tts_start_ms`.
2. Добавить tts-specific debug logs в модуле (время инициализации, генерации, исключения).
3. Контролировать деградации по queue/interrupt при включенном TTS.

## 7. Сверка с аудитом Grok: что подтвердилось и что нужно поправить

Подтвердилось:

1. Асинхронный event-driven пайплайн и модульная архитектура реально есть.
2. Voice interrupt/turn-taking существенно продвинуты.
3. STT + llama.cpp локально интегрированы и рабочие.

Нужно поправить в формулировках:

1. В текущем состоянии проекта TTS не «на 95% подключен», а инфраструктурно готов, но модуль отсутствует.
2. Эмоции есть как модуль `emotion_roberta`, но не активированы в текущем `config.yaml`.
3. Интеграции Discord/Twitch/VTS в этом репозитории сейчас отсутствуют как приложения.
4. UI пока не проигрывает аудио чанки, хотя backend их может стримить при наличии TTS.

## 8. Архитектурное решение на ближайший этап

1. Внедрять TTS только как отдельный operation plugin.
2. Не менять логику `response_pipeline` кроме минимальных guard/telemetry правок при необходимости.
3. Сначала добиться стабильного streaming TTS без `filter_audio`.
4. Потом включать `emotion_roberta` и аудио-фильтры по одному, с smoke/regression проверкой после каждого шага.
