# Архитектура Nira (актуальное состояние, стандарты и целевой TTS)

Дата актуализации: 2026-03-04  
Репозиторий: `C:\Nirmita`

## 1. Назначение

Nira это локальный асинхронный backend AI-персонажа с:
1. Централизованным application layer `JAIson`.
2. Job queue для response-пайплайна.
3. Immediate STT-path для быстрых голосовых прерываний.
4. Модульной операционной моделью (`stt`, `t2t`, `filter_*`, `tts`).
5. Наблюдаемостью через WebSocket события и runtime-метрики.

Цель текущего этапа: устойчивый и быстрый STT тракт с модульным подключением провайдеров и готовностью к отдельному TTS-плагину.

## 2. Фактическое состояние проекта

## 2.1 Реализовано и активно

1. Quart backend + WS сервер событий.
2. Очередь задач с отменой running/queued jobs и безопасной очисткой coroutine.
3. Голосовой immediate тракт: `POST /api/context/conversation/audio` обрабатывается вне общей очереди.
4. Ранний сигнал речи: `POST /api/context/conversation/speech_start`.
5. Активный STT выбирается через `stt_active_id` в `config.yaml`.
6. STT-провайдер по умолчанию: `sherpa_ru` через sidecar server (`ws://127.0.0.1:6106`).
7. T2T-провайдер: `llamacpp` (managed `llama-server` процесс, потоковый вывод).
8. Text filters: `filter_clean` + `chunker_sentence`.
9. UI отображает стрим текста, историю, pipeline runtime и метрики (`tps`, `ttft_ms`, `e2e_ttft_ms`, `tts_start_ms`, `e2e_tts_start_ms`).
10. Микрофонный клиент `hw-mic-client` с локальным VAD/RMS, pre-roll, barge-in и отправкой `turn_id/utterance_id/source_id`.

## 2.2 Реализовано в коде, но не включено в текущем профиле

1. `emotion_roberta` (text emotion filter).
2. `pitch` и `rvc` audio filters.
3. MCP manager и embedding operation.
4. Sherpa punctuation post-processing через Silero TE (`enable_te`), но по умолчанию отключено.

## 2.3 Пока отсутствует

1. Рабочая TTS operation (ветка `OpTypes.TTS` в `OperationManager.load_op()` пока возвращает `UnknownOpID`).
2. Активная `role: tts` запись в `configs/config.yaml`.
3. Воспроизведение `audio_chunk` в `nira-web`.
4. Полный контур долговременной памяти (vector DB + retrieval loop).
5. Приложения Discord/Twitch/VTube Studio в текущем репозитории.

## 3. Стандартизированные архитектурные контракты

## 3.1 Общий контракт Operation

1. Любая operation наследуется от `Operation`.
2. Обязательные этапы: `configure()`, `get_configuration()`, `_parse_chunk()`, `_generate()`.
3. Runtime интерфейс единый: `async for chunk in operation(payload)`.
4. Горячее переключение/перезагрузка операций: `/api/operations/reload`.

## 3.2 STT V2 контракт (принят как стандарт)

Вход:
1. `audio_bytes` (bytes PCM).
2. `sr`, `sw`, `ch`.
3. Опционально: `prompt`, `source_id`, `turn_id`, `utterance_id`, `speaker_id`, `input_timestamp_ms`.

Выход:
1. `text` (основной транскрипт).
2. `is_final` (bool).
3. `confidence` (float|None).
4. `provider`.
5. `source_id`, `turn_id`, `utterance_id`, `speaker_id`.
6. `stt_latency_ms`.
7. `transcription` как legacy alias для обратной совместимости.

Примечание:
1. `process_audio_immediate()` читает только `is_final=true`, partial-ready логика уже заложена в контракте.

## 3.3 Контракт sidecar lifecycle для STT

Стандарт управления внешним STT процессом:
1. `STTProcessRunner.ensure_running()`.
2. `STTProcessRunner.health()`.
3. `STTProcessRunner.ensure_healthy()`.
4. `STTProcessRunner.restart()`.

Параметры запуска стандартизированы в конфиге:
1. `process_autostart`.
2. `process_startup_retries`.
3. `process_startup_backoff_s`.
4. `process`-блок с провайдер-спецификой (`port`, `provider`, `model_dir`, hotwords и т.д.).

## 4. Ключевые тракты данных

## 4.1 Голосовой вход

`hw-mic-client`  
-> VAD/RMS сегментация + pre-roll + short-interrupt logic  
-> `speech_start`  
-> `audio` (base64 PCM + source/turn/utterance)  
-> `JAIson.submit_audio_immediate()` (backpressure-aware)  
-> `JAIson.process_audio_immediate()`  
-> `STT`  
-> intent/semantic classification (`stop`, `backchannel`, `continue`, `wake`, `emote`, `significant`)  
-> merge/debounce голосового хода  
-> постановка `context_conversation_add_text` и при необходимости `response`.

## 4.2 Текстовый вход

`POST /api/context/conversation/text`  
-> запись в prompter history  
-> `POST /api/response`  
-> job queue.

## 4.3 Тракт ответа

`response_pipeline()`:
1. Подготовка prompt/history.
2. Потоковый T2T.
3. Потоковая публикация текста + метрик TTFT/TPS.
4. FILTER_TEXT chain.
5. Условный TTS path (готов в пайплайне, но провайдер пока отсутствует).
6. Условный FILTER_AUDIO chain.
7. Финальная запись ассистент-ответа в историю.

## 5. STT модульность и переключаемость

## 5.1 Выбор активного STT

1. В `operations` можно держать несколько `role: stt`.
2. Активный выбирается по `stt_active_id`.
3. Если `stt_strict_active_id: true`, невалидный id блокирует старт (без fallback).
4. Reload выполняется через `/api/operations/reload`.

## 5.2 Реестр STT провайдеров

1. `utils.operations.stt.registry.load_stt_operation()` поддерживает:
   1. Built-in mapping.
   2. Динамический `entrypoint: "module.path:ClassName"`.
2. Добавление нового STT провайдера не требует правок `jaison.py`.
3. Core правило: новый STT обязан реализовать STT V2 контракт.

## 5.3 Sherpa ONNX как reference implementation

1. Поддерживает runtime process config + preset (`low_latency`, `balanced`, `noisy_room`).
2. Имеет recovery-поведение при ошибках канала (`timeout`, `restarting`, `unavailable`).
3. Логирует final метрики и публикует `stt_error`.
4. Hotwords проходят валидацию/санитизацию перед запуском sidecar.

## 6. Полировка VAD/STT фронтенда (внедрено)

В `hw-mic-client` стандартизированы:
1. Высококачественный ресемпл `polyphase` (с fallback `decimate`).
2. Программный микрофонный gain (`mic_gain_db`).
3. Консервативный AGC (`agc_enable`, `agc_target_rms`, `agc_max_gain_db`).
4. Soft limiter (`soft_limit`).
5. RMS/VAD параметры начала/удержания (`vad_start_rms`, `vad_hold_rms`, `vad_floor_rms`).
6. Короткий bridge (`rms_bridge_ms`) между слогами/микропаузами без роста TTFT.
7. Анти-фрагментация: reset gap-счетчика при `vad_active` или `rms_active`.

Результат целевого режима:
1. Сегментация не уходит в 12-секундные хвосты.
2. TTFT не деградирует до секунд.
3. Тихая речь остается распознаваемой без сильной потери темпа.

## 7. Наблюдаемость и эксплуатационные стандарты

## 7.1 Runtime endpoint

`GET /api/pipeline` включает:
1. `queue_size`, `status`, `current_job_id`.
2. `stt.immediate_active`, `stt.immediate_pending`, `stt_window_size`.
3. System stats (CPU/RAM/GPU).

## 7.2 STT backpressure и деградации

1. Ограничения: `stt_immediate_max_active`, `stt_immediate_max_pending`.
2. Политики: `drop_oldest`, `drop_latest`, `merge_latest`.
3. События деградаций в WS: `stt_status` (`timeout`, `unavailable`, `restarting`, backpressure states).
4. Локальное накопление метрик и трасс в `logs/stt_events.jsonl`.

## 7.3 Диалоговые policy

1. `speech_start_interrupt_mode: soft|hard`.
2. `interrupt_speaker_policy: any|same_only|same_or_unknown`.
3. `voice_merge_window_ms`, `response_debounce_ms`, `response_min_quiet_ms_after_speech_start`.
4. Wake aliases и short-emote правила.

## 8. Известные ограничения и правила качества

1. Пунктуация STT сейчас не гарантируется.
2. Причина: punctuation post-processing в Sherpa отключен по умолчанию (`enable_te: false`) и требует локальных Silero TE весов.
3. TTS слой архитектурно подготовлен, но не подключен.
4. UI пока не воспроизводит `audio_chunk`.
5. Долгосрочная память и voice-id расширения остаются отдельным этапом.

## 9. Целевой TTS стандарт (для следующего этапа)

## 9.1 Принцип

1. TTS внедряется только как отдельная operation plugin.
2. `jaison.py` не должен получать provider-specific TTS логики.

## 9.2 Контракт TTS

Вход:
```json
{ "content": "текст", "emotion": "опционально" }
```

Выход:
```json
{ "audio_bytes": "bytes", "sr": 24000, "sw": 2, "ch": 1 }
```

Требования:
1. Стриминг чанками без блокировки event loop.
2. Стабильный audio format в пределах сессии.
3. Логи и метрики времени до первого аудио чанка.

## 10. Стандартизированный runbook изменения STT

1. Добавить/изменить STT блок в `operations` (`id`, `entrypoint`, `process`).
2. Выставить `stt_active_id`.
3. При необходимости включить `stt_strict_active_id: true`.
4. Выполнить `/api/operations/reload` или перезапуск backend.
5. Проверить `GET /api/operations` и `GET /api/pipeline`.
6. Прогнать smoke/regression (`src/tools/stt_smoke.py`, `src/tools/stt_regression.py`).
