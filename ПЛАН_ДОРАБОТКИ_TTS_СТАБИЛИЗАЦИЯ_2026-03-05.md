# ПЛАН_ДОРАБОТКИ_TTS_СТАБИЛИЗАЦИЯ_2026-03-05

## Контекст
Текущий тракт TTS в целом быстрый и рабочий, но по живому диалогу есть 7 системных проблем: стартовые щелчки/шум, плавающий TTFAudio, нестабильный тембр, артефакты на длинных репликах.

## Что подтверждено логами
- На длинных ответах есть переполнение буфера выхода с дропом аудио:
  - `Buffer overflow: dropped ... (queue=.../1152000)` в `logs/backend_console_2026-03-05.log`.
- Есть большие паузы между чанками TTS даже при нормальном RTF:
  - `max_gap_ms` периодически 500-725ms (`logs/2026-03-05.log`).
- В начале некоторых реплик первые чанки почти тишина/шум:
  - `PCM chunk #1/#2/#3 rms` очень низкий (до `0.0001`) в `logs/backend_console_2026-03-05.log`.
- Клон-голос сейчас собран в самом нестабильном режиме:
  - `x_vector_only_mode=True`, `ref_text=empty` (`logs/qwen3_tts_server.log`).
- В конфиге сейчас `ref_audio_path = C:\Nirmita\kuklina-1.wav`, что может не совпадать с целевым голосом персонажа.

## Разбор по пунктам и пути решения

### 1) Щелчок/шум и немая пауза в начале фразы
Вероятные причины:
- Стартовые чанки содержат низкоамплитудный шум/тишину, но воспроизведение уже начато.
- Недостаточная «де-клик» обработка начала потока.

Пути решения:
- Ввести `first-voiced gate` (RMS-порог только для старта реплики): не выпускать первые чанки, пока RMS ниже порога, но не дольше ограничителя (например 250-400ms).
- Добавить мягкий старт/стоп (fade in/out) и DC-block/high-pass на очень низких частотах.
- Добавить метрику `first_voiced_ms` отдельно от `tts_start_ms`.

### 2) TTFAudio слишком высокий/плавающий
Вероятные причины:
- `TTFAudio e2e` включает не только TTS provider, но и задержку до момента, когда есть первый реально пригодный чанк.
- Для длинных ответов T2T + фильтры/очереди дают большой разброс.

Пути решения:
- Разделить метрики в UI и логах:
  - `TTFAUDIO_PROVIDER` (уже есть)
  - `TTFAUDIO_E2E_RAW` (уже есть)
  - `TTFVOICE_E2E` (новая: до первого «озвученного» чанка).
- Перевести текстовый тракт в гибрид:
  - первый кусок/первая фраза отправляется в TTS раньше,
  - хвост — батчами (без агрессивной нарезки).
- Ввести target-профили длины ответа (ограничить чрезмерно длинные реплики LLM).

### 3) Искажение голоса в начале каждой реплики
Вероятные причины:
- Нестабильное состояние голосового промпта на старте при `x_vector_only_mode=True` и пустом `ref_text`.
- Резкий вход без достаточной стабилизации первых кадров.

Пути решения:
- Перейти на полноценный voice-clone prompt:
  - `x_vector_only_mode = false`
  - заполненный `ref_text` (точная транскрипция референса).
- Добавить стартер-обработку первых кадров (onset smoothing + gate).
- Проверить `overlap_samples` > 0 (например 256/512) в A/B на артефакты стыков.

### 4) Голос «плавает» (то мужской, то женский)
Вероятные причины:
- Клон по x-vector без текста хуже фиксирует идентичность.
- Возможное несоответствие целевого референса (`kuklina-1.wav` vs ожидаемый голос персонажа).

Пути решения:
- Зафиксировать единый эталонный референс (один WAV, чистый, 10-20с, стабильный тембр).
- Обязательно задать `ref_text` и отключить `x_vector_only_mode`.
- Зафиксировать стиль через системную текстовую нормализацию (пунктуация/длина фраз/эмоциональность), чтобы не дергать тембр.

### 5) На длинных фразах чанки «наезжают»/прыжки вперед
Вероятные причины:
- Прямо подтверждено: переполнение буфера аудио-выхода и дроп входящих данных (`Buffer overflow`).
- Длинная генерация при RTF<1 создает аудио быстрее реального времени, очередь переполняется.

Пути решения:
- Ввести защиту на уровне контента:
  - ограничение максимальной длины одной реплики для озвучки,
  - принудительная сегментация очень длинного ответа.
- Ввести queue-aware backpressure:
  - при достижении high-watermark ограничивать подачу новых аудио-чанков.
- Подобрать `max_buffer_ms` адаптивно (не бесконечно большой, иначе растет латентность/эхо бардж-ина).

### 6) Фоновый шум, особенно в конце
Вероятные причины:
- Низкоамплитудный хвост модели + отсутствие tail-trim/noise gate.

Пути решения:
- Tail trim по RMS + короткий fade-out при завершении потока.
- Легкий noise gate только на «хвосте» (чтобы не ломать тихую речь внутри фразы).
- Опционально: простой post-filter (high-pass/denoise-light) как FILTER_AUDIO.

### 7) Управление ударениями
Ограничение:
- В текущем Qwen3-TTS нет «из коробки» полноценной SSML-экосистемы как у некоторых коммерческих TTS.

Пути решения без глубокой переделки:
- Словарь исключений перед TTS (preprocess):
  - `слово -> словоформа с ударением` (например `за́мок/замо́к`, `всё/все`),
  - поддержка `ё`.
- Локальная разметка ударения в тексте (A/B проверить, как модель лучше понимает):
  - combining acute (`а́`),
  - дефисная подсказка по слогам для сложных слов.
- Добавить пользовательский YAML-лексикон ударений в конфиг.

## Приоритет внедрения

### Этап 0 (быстрый, 0.5-1 день)
- Зафиксировать voice clone:
  - правильный `ref_audio_path`,
  - `ref_text` заполнить,
  - `x_vector_only_mode=false`.
- Добавить лог-метрики:
  - `first_voiced_ms`,
  - `leading_silence_ms`,
  - `overflow_count`, `dropped_bytes_total`.

### Этап 1 (1-2 дня)
- Реализовать onset/tail обработку:
  - start gate + fade-in,
  - tail trim + fade-out.
- Стабилизировать TTFAudio:
  - гибридный ранний старт TTS по первому текстовому куску,
  - ограничение сверхдлинных озвучиваемых ответов.

### Этап 2 (2-3 дня)
- Queue-aware backpressure между TTS и audio_out.
- A/B по `overlap_samples` и window-параметрам с одинаковым датасетом фраз.
- Добавить лексикон ударений и pre-TTS нормализатор.

## Критерии приемки
- Нет `Buffer overflow` в 15-мин диалоге.
- `first_voiced_ms` p95 <= 1200ms (короткие ответы), <= 2200ms (длинные).
- `max_gap_ms` p95 <= 350ms.
- Стартовый щелчок/шум субъективно исчезает в большинстве реплик.
- Тембр стабилен между репликами (без явных переходов «м/ж»).
- На длинных ответах нет прыжков вперед/наезжания чанков.

## Важная ремарка по текущему состоянию
Сейчас при длинных ответах узкое место уже не в скорости генерации модели (RTF часто < 1), а в управлении стартом/буфером/клон-параметрами и backpressure в воспроизведении.
