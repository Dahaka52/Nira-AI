# План внедрения TTS как отдельного подключаемого модуля

Дата: 2026-03-04  
Проект: `C:\Nirmita`

## 1. Цель

Подключить TTS в существующий пайплайн Nira без ломки текущей архитектуры, именно как plug-in операцию (`role: tts`), с поддержкой стриминга аудио чанков и корректной работы interrupt/queue логики.

## 2. Границы работ

Входит:

1. Новый модуль `src/utils/operations/tts/<provider>.py`.
2. Регистрация провайдера в `OperationManager.load_op()`.
3. Конфигурация в `configs/config.yaml`.
4. Smoke + e2e проверка через API/WebSocket.
5. Обновление документации.

Не входит в первый проход:

1. Полноценный voice cloning/RVC-тюнинг.
2. Долгосрочная память.
3. Отдельные интеграции Discord/Twitch/VTS.

## 3. Definition of Done

1. `POST /api/operations/reload` загружает TTS без ошибки.
2. `POST /api/response` с `include_audio=true` выдаёт `audio_chunk` события.
3. Нет регрессий в stop/interrupt и job cancellation.
4. Метрики `tts_start_ms` и `e2e_tts_start_ms` приходят в WS.
5. Повторный запуск и reload работают стабильно.

## 4. Пошаговый план

### Шаг 0. Зафиксировать baseline

Сделать:

1. Убедиться, что текущий `main`/ветка чистая (`git status`).
2. Снять baseline логи latency без TTS (TTFT/TPS/queue behavior).
3. Сохранить текущий `configs/config.yaml` как rollback copy.

Критерий:

1. Есть baseline показатели и точка отката.

### Шаг 1. Выбрать TTS-провайдер и runtime параметры

Сделать:

1. Зафиксировать `provider_id` (например, `qwen3`).
2. Определить `sample_rate`, `channels`, `sample_width`, `device`, `voice`.
3. Уточнить формат стрима модели: raw PCM или float, и где выполнять конвертацию.

Критерий:

1. Есть согласованный контракт входа/выхода для `_generate()`.

### Шаг 2. Подготовить зависимости и окружение

Сделать:

1. Добавить нужный пакет(ы) в зависимости проекта.
2. Проверить импорт библиотеки из текущего `.conda` окружения.
3. Проверить доступность CUDA для модели (если выбран GPU режим).

Критерий:

1. Модуль можно импортировать без runtime ошибок.

### Шаг 3. Создать TTS-операцию

Файл:

`src/utils/operations/tts/qwen3.py` (или другой провайдер по выбранному id)

Сделать:

1. Наследовать класс от `TTSOperation`.
2. Реализовать `configure()` со строгой валидацией параметров.
3. Реализовать `start()` с ленивой/безопасной инициализацией модели.
4. Реализовать `close()` с освобождением ресурсов.
5. Реализовать `_generate(content, emotion=None, **kwargs)` со стримингом чанков.
6. На выходе каждого чанка отдавать строго: `audio_bytes`, `sr`, `sw`, `ch`.

Критерий:

1. `OperationManager.use_operation(OpRoles.TTS, ...)` возвращает валидные аудио чанки.

### Шаг 4. Зарегистрировать TTS в OperationManager

Файл:

`src/utils/operations/manager.py`

Сделать:

1. В ветке `case OpTypes.TTS` добавить `if op_id == "qwen3": ... return Qwen3TTS()`.
2. Оставить `UnknownOpID("TTS", op_id)` для неподдерживаемых id.

Критерий:

1. Reload операций проходит без `UnknownOpID` для нового TTS.

### Шаг 5. Добавить TTS в конфиг

Файл:

`configs/config.yaml`

Сделать:

1. Добавить блок:

```yaml
- role: tts
  id: qwen3
  model_id: "..."
  device: "cuda"
  sample_rate: 24000
  voice: "nira"
  stream_chunk_ms: 80
```

2. Временно не включать `filter_audio`, пока не подтверждён базовый TTS поток.

Критерий:

1. `POST /api/operations/reload` поднимает `tts: qwen3` в `/api/operations`.

### Шаг 6. Smoke-тест standalone TTS

Сделать:

1. Вызвать `/api/operations/use` для роли `tts` с payload `{content: "Тест"}`.
2. Проверить через WS, что идут чанки с `audio_bytes` и валидными `sr/sw/ch`.
3. Проверить ошибки на пустом контенте и длинном контенте.

PowerShell пример:

```powershell
Invoke-RestMethod -Method Post -Uri "http://127.0.0.1:7272/api/operations/use" -ContentType "application/json" -Body (@{
  role = "tts"
  id = "qwen3"
  payload = @{ content = "Привет, это тест TTS" }
} | ConvertTo-Json -Depth 6)
```

Критерий:

1. Модуль работает независимо от полного response pipeline.

### Шаг 7. E2E тест полного response pipeline с аудио

Сделать:

1. Добавить пользовательский текст через `/api/context/conversation/text`.
2. Запустить `/api/response` с `include_audio=true`.
3. Проверить WS последовательность: text chunks -> audio_chunk -> finish.
4. Проверить наличие `tts_start_ms` и `e2e_tts_start_ms`.

Критерий:

1. Полный цикл text->tts->ws стабилен минимум на 20 последовательных запросах.

### Шаг 8. Проверить совместимость с interrupt/barge-in

Сделать:

1. Во время генерации ответа дать голосовую реплику с `speech_start`.
2. Проверить, что активный response корректно отменяется/останавливается.
3. Проверить, что новая реплика уходит в следующий user turn и генерирует новый ответ.

Критерий:

1. Нет подвисших jobs и нет «залипания» speaking/thinking состояния.

### Шаг 9. Подключить эмоции к TTS

Сделать:

1. Включить `emotion_roberta` в `operations` перед `chunker_sentence` или после (по выбранной логике).
2. В TTS модуле аккуратно маппить `emotion` в стиль синтеза.
3. Добавить fallback: если emotion отсутствует, обычная нейтральная речь.

Критерий:

1. На репликах с разным тоном меняется просодия без деградации стабильности.

### Шаг 10. Подключить audio filters по одному

Сделать:

1. Сначала `pitch`, затем при необходимости `rvc`.
2. После каждого включения прогнать short stress test.
3. Проверить, что формат аудио (`sr/sw/ch`) остаётся консистентным до WS.

Критерий:

1. Фильтры не ломают поток и не вызывают заметный рост задержки старта речи.

### Шаг 11. Нагрузочное тестирование

Сделать:

1. 30-50 запросов подряд с `include_audio=true`.
2. Сценарий с частыми прерываниями (stop/backchannel/continue).
3. Мониторинг CPU/RAM/GPU через `/api/pipeline` и логи.

Критерий:

1. Нет утечек, падений процесса, и деградации от запроса к запросу.

### Шаг 12. Веб-клиент и аудио-вывод

Сделать:

1. Решить цель вывода: только WS-stream, либо локальный playback/виртуальный кабель.
2. Если нужен playback в `nira-web`, добавить обработку события `audio_chunk` и буфер аудио.
3. Если нужен системный вывод в виртуальный микрофон, реализовать отдельный output-адаптер вне `JAIson` ядра.

Критерий:

1. Аудио реально слышно в целевом месте (UI и/или виртуальный кабель).

### Шаг 13. Документация и rollback

Сделать:

1. Обновить `АРХИТЕКТУРА проекта Нира.md` и эксплуатационные заметки.
2. Добавить быстрый rollback сценарий: убрать `role: tts` из конфига + reload.
3. Зафиксировать known issues и рабочие параметры модели.

Критерий:

1. Новый TTS можно отключить без изменений кода за 1 минуту.

## 5. Контрольные риски и меры

1. Риск: блокировка event loop тяжёлой генерацией.  
Мера: chunked streaming и вынос тяжёлых вызовов в thread/executor.

2. Риск: нестабильный аудио формат между чанками.  
Мера: единый `sr/sw/ch` policy на уровне модуля.

3. Риск: рост latency из-за `filter_audio`.  
Мера: включать фильтры последовательно и мерить `tts_start_ms`.

4. Риск: регресс interrupt логики.  
Мера: отдельный regression suite сценариев stop/continue/wake.

## 6. Минимальный чеклист релиза

1. [ ] TTS модуль загружается через `operations/reload`.
2. [ ] `response(include_audio=true)` отдаёт аудио чанки.
3. [ ] Метрики TTS старта приходят в WS.
4. [ ] Stop/interrupt не ломается.
5. [ ] Есть rollback через конфиг без правок Python-кода.
