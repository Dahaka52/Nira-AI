# План Решения TTS Проблем (после внешнего аудита)

Дата: 2026-03-04  
Контекст: текущий `qwen3` sidecar на Win11, локально, streaming в WS + `hw-audio-out-client`.

## 1. Что полезного взято из внешнего аудита

Подтверждено и полезно:
- Нужна строгая диагностика: отличить проблему генерации (сам WAV уже рваный) от проблемы транспорта/плеера.
- Возможны два разных класса дефектов:
  - щелчки/ломаная огибающая на границах чанков;
  - длинные паузы из-за underrun (генерация медленнее воспроизведения).
- Требуется jitter/prebuffer (уже частично внедрено, но нужно тюнинговать).
- Нужно избегать лишней нарезки/overhead при WS-передаче аудио.
- Нужен fallback-план (второй TTS модуль) на случай нестабильности Qwen3 streaming.

Сомнительно/не берем как факт без локальной проверки:
- Универсальные утверждения про «обязательный compile+cuda-graphs» для всех систем.
- Часть ссылок и цифр latency/RTF без воспроизводимого локального бенчмарка.

## 2. Что подтверждено локальными логами

- `qwen3_tts_server.log`:
  - frequent `cudagraph partition due to dynamic shape ops`;
  - warmup очень долгий (десятки секунд);
  - `first-audio latency` скачет (примерно 3s, 4s, 7s+).
- `backend`:
  - много RESPONSE jobs отменяется `Smart Barge-in` до завершения TTS;
  - в отдельных сессиях `TTS operation qwen3` занимает 10–30+ секунд.
- `audio_out`:
  - регулярный `Rebuffering: 0/...` => буфер пустеет, нет стабильного притока PCM.

Вывод: сейчас одновременно присутствуют:
1) низкая/нестабильная скорость генерации чанков;  
2) частые прерывания RESPONSE из STT/barge-in;  
3) лишний overhead передачи на WS уровне.

## 3. Цели исправления

- TTFS (first audible speech) <= 1.5s в типовой короткой реплике.
- Без длинных пауз внутри одной реплики (не более редких микропауз < 150ms).
- Отсутствие «залипания»/бесконечного ожидания чанков.
- Управляемый barge-in: не рушить ответ ложными срабатываниями.

## 4. План внедрения

## Этап A — Быстрая стабилизация (внедряем сразу)
1. Убрать лишний WS overhead:
- отправлять `audio_chunk` одним сообщением (без дополнительной нарезки base64 в `jaison.py`).

2. Уменьшить риск «зависаний» Qwen3:
- временно дефолт `use_compile=0` для sidecar (оставить compile как опцию A/B).
- сохранить `flash_attention_2` в Py3.12 окружении.

3. Снизить риск авто-отмены RESPONSE:
- ввести настраиваемый порог `barge_in_min_non_filler_words` (по умолчанию 2, для теста поставить 4).
- при ложных срабатываниях дать конфиг-режим, где interrupt только по stop-командам.

4. Снизить latency до первого звучания:
- переключить `chunker_sentence` в `sentence` (вместо `full`) для более ранней подачи первого TTS чанка.

Критерий этапа A:
- реплика 1–2 предложения проигрывается без полной «тишины после первых слов».

## Этап B — Диагностика «генерация vs транспорт» (обязательно)
1. Добавить debug-режим записи:
- склейка входящих PCM от sidecar в локальный WAV на 1 запрос.

2. Добавить метрики чанков:
- количество, средний интервал между чанками, максимум интервала, суммарная аудио-длительность.

3. Сверка:
- если WAV уже рваный -> проблема в генерации/склейке чанков;
- если WAV ровный, а realtime рваный -> проблема транспорта/плеера/буфера.

## Этап C — Качество стыков чанков
1. Если останутся щелчки/ломаный тон:
- добавить опциональный Hann crossfade на границе chunk-to-chunk (5–15ms) на стороне playback или sidecar emit layer.

2. Провести A/B:
- `overlap_samples = 0` vs небольшое overlap + crossfade.

## Этап D — Производительность и форки
1. A/B профили sidecar:
- profile `stable`: compile=0;
- profile `fast`: compile=1.

2. Если `stable` все еще плох:
- отдельная ветка интеграции `dffdeeq/reku` runtime (без ломки текущего pipeline API).

## Этап E — Fallback
1. Добавить второй TTS operation (`tts_active_id` switch) как backup.
2. Переключение только конфигом, без переписывания pipeline.

## 5. Порядок выполнения прямо сейчас

1) Этап A.1/A.2/A.3/A.4 (код + конфиг).  
2) Этап B (WAV/chunk metrics).  
3) Повторный живой прогон и сравнение логов.  
4) Решение по Этапу C/D на основе фактов.

## 6. Статус внедрения (текущая итерация)

Сделано:
- `jaison.py`: убрана дополнительная нарезка base64 (`chunk_buffer`) при WS-отправке `audio_chunk` (1 событие на 1 PCM-чанк).
- `jaison.py`: добавлен конфиг `microphone.barge_in_min_non_filler_words` (используется только для прерывания активного ответа).
- `config.yaml`:
  - `chunker_sentence.mode = "sentence"`
  - `operations.tts[qwen3].process.use_compile = 0` (stable baseline для A/B)
  - `microphone.barge_in_min_non_filler_words = 4`
- `qwen3.py`: добавлена телеметрия стрима в backend-лог:
  - `first_chunk_ms`
  - `max_gap_ms`
  - `chunks/bytes/total_ms/text_len`

Что проверить в следующем живом прогоне:
- Появился ли устойчивый поток (исчезли ли частые `Rebuffering: 0/...`).
- Значения `Qwen3 stream stats`:
  - если `max_gap_ms` большой (сотни/тысячи мс) -> узкое место генерации sidecar;
  - если `max_gap_ms` низкий, а звук рваный -> проблема на playback/route уровне.

Обновление (последняя итерация):
- `hw-audio-out-client` получил anti-stall rebuffer:
  - `rebuffer_max_wait_ms` (новый параметр) и динамический порог возобновления.
  - цель: не зависать в бесконечном `Rebuffering` при редких/неровных чанках.
- `audio_output` в конфиге:
  - `prebuffer_ms: 120`
  - `rebuffer_ms: 120`
  - `rebuffer_max_wait_ms: 220`
- Профиль Qwen3 ужат под более частую эмиссию:
  - `emit_every_frames: 4`
  - `decode_window_frames: 64`
  - `first_chunk_emit_every: 2`
  - `first_chunk_decode_window: 32`
  - `first_chunk_frames: 32`
- `chunker_sentence.mode` снова выставлен в `full` для уменьшения повторных first-chunk задержек внутри одной реплики.
