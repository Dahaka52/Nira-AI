# План Улучшений Скорости TTS По Ресерчу (2026-03-04)

## 1) Подтвержденные факты по текущему состоянию

- По живым логам узкое место в генерации, а не в маршрутизации звука:
  - `Qwen3 stream stats`: `rtf` стабильно около `3.7..4.6` (хуже realtime).
  - `first_chunk_ms`: часто `1.4..3.1s`.
  - `max_gap_ms`: часто `1.1..1.4s` (это и вызывает дырки в воспроизведении).
- `AUDIO_OUT Rebuffering` в этих условиях является следствием (плеер ждет новые PCM), а не первопричиной.
- В части запусков были длинные/аномальные хвосты (runaway) на voice-clone; уже добавлены динамические лимиты `max_frames`, но throughput все равно недостаточный.
- Текущий sidecar запущен с `use_compile=0` (стабильнее, но медленнее).

## 2) Что полезного подтверждено из форков/ресерча

Проверено по локальным копиям:
- `.tmp_forks/Qwen3-TTS-streaming-dffdeeq/README.md`
- `.tmp_forks/faster-qwen3-tts/README.md`
- `.tmp_forks/Qwen3-TTS-Openai-Fastapi/README.md`
- `.tmp_forks/Qwen3-TTS-Openai-Fastapi/OPTIMIZATION_GUIDE.md`

Практически полезно:
- `dffdeeq`: реальный streaming API + заявлен ускоренный runtime, Python 3.12, акцент на `emit_every_frames`/`decode_window_frames`.
- `faster-qwen3-tts`: отдельный runtime на CUDA Graph/Static Cache, сильный фокус именно на снижении dispatch overhead.
- `Openai-Fastapi optimized backend`: готовые инженерные паттерны для прод-режима (compile, TF32, warmup, voice cache, streaming PCM).

Важно:
- В форках используется разная трактовка RTF.
- В твоих логах RTF считается как `total_s / audio_s` (меньше лучше).
- В `faster-qwen3-tts` таблице указано обратное (`RTF > 1` лучше).
- Сравнивать можно только после нормализации метрики.

## 3) Что из ресерча не берем как факт без локальной проверки

- Любые универсальные обещания "RTF < 1 из коробки" без учета твоего точного режима (`voice_clone`, Win11, конкретные параметры чанков).
- Перенос цифр с RTX 4090/5090/3090 напрямую на твой pipeline без A/B на твоем железе.
- Утверждения про vLLM-Omni как готовый онлайн-serving путь для этой конкретной задачи без отдельного интеграционного теста.

## 4) Главная гипотеза

Ключевая проблема сейчас: низкая эффективная пропускная способность decode-цикла в текущем runtime (`qwen_tts` sidecar, compile disabled), из-за чего поток PCM приходит реже, чем потребляется плеером.

Следствие:
- буфер постоянно проваливается в rebuffer,
- фразы звучат кусками,
- при длинных репликах поток "сыпется" заметнее.

## 5) Пошаговый план внедрения (приоритетный)

## Этап A. Зафиксировать эталонные метрики (без изменений кода)

Цель: получить базу для сравнения.

Снимаем для 20 одинаковых реплик:
- `first_chunk_ms`
- `max_gap_ms`
- `rtf`
- `audio_s`
- частоту `Rebuffering` в минуту
- среднюю длину непрерывного звучания (сек)

Критерий: единый CSV/JSON baseline перед ускорением.

## Этап B. Точечный speed-профиль внутри текущего sidecar

Цель: выжать максимум из текущего стека до миграции runtime.

1. A/B `compile` только в профиле ускорения:
- `use_compile=1`
- `allow_compile_voice_clone=1`
- `compile_mode=max-autotune` (для speed-профиля)
- `compile_use_cuda_graphs=1` (если не падает рантайм)

2. Укрупнить порции декода для снижения overhead:
- тест-сетка:
  - `emit_every_frames`: `4`, `6`, `8`
  - `decode_window_frames`: `64`, `80`
  - `first_chunk_emit_every`: `2`, `3`
  - `first_chunk_decode_window`: `32`, `48`
- выбирать по минимуму `max_gap_ms`, а не только по `first_chunk_ms`.

3. Увеличить сетевой квант чтения:
- `sidecar_read_chunk_bytes`: `8192` и `16384` A/B.

Критерий прохождения этапа:
- `max_gap_ms <= 350ms` на коротких фразах;
- `rtf <= 1.3` на коротких/средних фразах;
- заметное снижение частоты `Rebuffering`.

## Этап C. Подключить альтернативный runtime как backend, не меняя контракт core

Цель: если Этап B не дает realtime, меняем двигатель, не ломая пайплайн.

1. Ввести `tts_runtime_backend`:
- `qwen_sidecar_current` (текущий)
- `qwen_dffdeeq`
- `qwen_faster`

2. Реализовать адаптеры с тем же контрактом ответа:
- streaming PCM bytes
- заголовки `sr/sw/ch`
- поддержка `voice_clone`

3. Прогнать тот же baseline-набор реплик и сравнить с этапом A/B.

Критерий:
- выбрать backend с лучшим `max_gap_ms` и устойчивым качеством без артефактов.

## Этап D. Довести playback под медленный/неровный поток

Цель: убрать остаточные щелчки и "хвостовой" шум.

1. Оставить conservative playback:
- без агрессивного fade по чанкам (`chunk_fade_ms=0`).
- контролируемый `prebuffer/rebuffer` без микрорезюма.

2. Добавить idle-guard:
- если поток завершен и >N секунд новых чанков нет, принудительно завершать текущий буферный цикл и сбрасывать хвост.

3. Добавить явную телеметрию плеера:
- underrun count
- суммарное время в rebuffer
- средняя длина непрерывного аудио-сегмента.

Критерий:
- отсутствие бесконечного `Rebuffering` в idle;
- отсутствие "писка/зацикленного хвоста" после конца реплики.

## 6) Целевые KPI после оптимизации

- `first_chunk_ms <= 900ms` (короткие фразы).
- `max_gap_ms <= 250ms` (короткие и средние реплики).
- `rtf <= 1.0` для типовых реплик ассистента.
- `Rebuffering` не чаще 1 события на 20 секунд непрерывной речи.

## 7) Порядок выполнения на практике

1. Этап A (baseline).
2. Этап B (speed-профиль текущего sidecar).
3. Если KPI не достигнуты, сразу Этап C (подключение `dffdeeq` runtime через адаптер).
4. После выбора backend — Этап D (финальная полировка playback).

## 8) Короткий вывод

Сейчас упираемся не в "кабель/войсметр", а в генерацию PCM-потока (большие межчанковые разрывы).  
Путь решения: сначала форсировать throughput (compile/cudagraph + параметры декода), затем при необходимости перейти на более быстрый runtime (`dffdeeq`/`faster`) с сохранением текущей архитектуры Nira.

## 11) Итерация после лога 20:47–20:48 (почему фраза недоозвучивается)

Найдено:
- в `voice_clone` почти всегда шло ограничение `max_frames=80` (видно в логах `Qwen3 voice-clone limits`),
- это ≈ 6.4 сек аудио (12Hz), поэтому длинные ответы обрываются после первых слов,
- при `chunker_sentence.mode=full` TTS получал длинный кусок целиком и чаще упирался в этот лимит.

Внесено:
- `chunker_sentence` переведен в `sentence` режим в конфиге;
- добавлен `max_chars` и разрез длинных предложений в `src/utils/operations/filter_text/chunker_sentence.py`;
- увеличены лимиты voice-clone:
  - `max_frames: 160`,
  - `dynamic_max_frames_cap: 160`,
  - `max_text_chars: 90` (чтобы не гонять слишком длинный chunk за один TTS-запрос).
- ограничена длина T2T-ответа (`t2t.max_length: 90`) для удержания realtime-профиля.

Ожидаемый эффект:
- меньше обрывов «после пары слов»;
- меньше длинных TTS-запросов;
- более ровное пофразовое озвучивание, даже при RTF>1.
